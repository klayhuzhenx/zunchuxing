## 一、审查目标

## 三、审查方法

### 3.1 审查流程

```
需求文档 §X 章节
    │
    ▼
┌─────────────────────────┐
│ Phase 1: 需求-代码映射   │  找出每个需求模块对应的代码文件
└───────────┬─────────────┘
            │
            ▼
┌─────────────────────────┐
│ Phase 2: 逐项对比审查    │  按字段/流程/交互/状态/校验 逐项检查
└───────────┬─────────────┘
            │
            ▼
┌─────────────────────────┐
│ Phase 3: 差异分类标注    │  缺失 → 🔴 | 偏差 → 🟡 | 多余 → 🔵 | 一致 → ✅
└───────────┬─────────────┘
            │
            ▼
┌─────────────────────────┐
│ Phase 4: 选择性修复      │  根据优先级决定修复/保留
└─────────────────────────┘
```

### 3.2 审查维度

每个需求模块的审查覆盖以下维度：

| 维度 | 检查内容 |
|---|---|
| **页面/路由** | 页面是否存在、路径是否正确 |
| **字段完整性** | 必填字段、校验规则、错误提示是否齐全 |
| **交互流程** | 按钮状态、弹窗、跳转、loading/empty/error 状态是否完整 |
| **业务规则** | 状态流转、计算逻辑、权限控制是否符合需求 |
| **数据展示** | 列表字段、详情信息、筛选条件是否匹配 |
| **视觉一致** | 品牌色、字体、图标、栅格是否符合设计规范 |

### 3.3 差异分类

| 标记 | 含义 | 处理原则 |
|---|---|---|
| ✅ | 与需求一致 | 无需修改 |
| 🔴 缺失 | 需求有但代码没有 | **必须补全** |
| 🟡 偏差 | 代码有但实现与需求不同 | **评估后调整** — 若代码实现优于需求则保留代码并更新需求；否则修改代码 |
| 🔵 多余 | 代码有但需求没有 | **评估后决定** — 属于MVP之外的合理扩展保留；否则标记待清理 |
| ⚪ 待确认 | 需求描述模糊或代码意图不明 | 标记待与产品/业务确认 |

## 九、修复优先级规则

```
P0 (阻塞)  → 核心业务流程缺失，影响MVP交付（如支付、下单、登录）
P1 (重要)  → 关键字段/校验缺失，影响数据正确性或用户体验
P2 (一般)  → 交互细节偏差、视觉不一致、次要功能缺失
P3 (可延后) → 优化建议、多余功能清理、文档更新
```

## 十一、工作约定

1. **逐Stage执行**：完成一个Stage的审查+修复后再进入下一个
2. **审查优先**：先完整审查输出报告，再根据报告选择性修复
3. **需求为准**：当代码实现与需求文档冲突时，以需求文档为准（除非确认需求已过时）
4. **保留决策记录**：每次"偏差"或"多余"的处理决定记录在报告中
5. **先读后改**：修改代码前必须先用Read工具阅读目标文件
6. **保持风格一致**：修改代码时匹配周围代码风格（命名、缩进、注释密度）
