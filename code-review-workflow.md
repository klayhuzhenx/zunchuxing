# 尊出行 · 三端代码审查工作流（C 模式）

> 版本：V1.0 | 日期：2026-06-10 | 基于 superpowers 技能组合

---

## 一、模式选择

经过实践验证，选用 **C 模式（折中）**：

| 阶段 | 方式 | 原因 |
|---|---|---|
| **比对阶段** | 派独立 subagent 审查 | 避免自我确认偏见，获取独立视角 |
| **修复阶段** | 主对话批量改 | 效率最高，同一文件多项修改合并 |
| **验证阶段** | `tsc --noEmit` | 快速静态检查 |

---

## 二、使用的 superpowers 技能

| 技能 | 用途 | 调用方式 |
|---|---|---|
| `superpowers:requesting-code-review` | 审查方法论——派独立 subagent、分级反馈 | `Skill` 工具激活，取其方法论 |
| `superpowers:systematic-debugging` | 修复方法论——根因分析、单变量验证 | `Skill` 工具激活，取其四阶段流程 |

## 三、完整工作流（6 步）

### Step 1 — 读取需求 + 代码

```
并行执行：
  ① Read 需求文档对应章节（如 §4-§5）
  ② Bash 列出代码文件结构
  ③ Read 全部相关代码文件
```

### Step 2 — 派独立 Reviewer

```
Agent 工具 → subagent_type: "general-purpose"
  prompt 中包含：
    - 需求文档路径 + 章节范围
    - 代码文件路径（一一对应关系）
    - 审查维度说明（字段/流程/交互/校验/状态）
    - 输出格式要求（🔴缺失/🟡偏差/🔵多余 + P0-P3）
    - 字数限制（1200-1500 字）
```

**Subagent 指令模板：**
```
你是独立代码审查员。严审前端代码，找出与需求规格的差异。

# 审查范围
需求文档：`xxx需求规格说明.md`  §X-§Y
代码目录：`xxx-app/src/pages/xxx/`

# 审查方法
按需求文档每章字段/流程/列表/弹窗/操作交互/状态规则逐项比对代码。

# 输出格式
| 编号 | 需求条目 | 类型 | 说明 | 优先级 |
P0=核心流程缺失；P1=关键字段/校验；P2=交互细节；P3=优化建议
控制在 1200 字内，只列差异不复述原文。
```

### Step 3 — 交叉验证 + 输出报告

```
我同时已读完代码 → 对照 subagent 报告交叉验证：
  - 确认 P0 项真实存在（如 txnStatusMap 未定义）
  - 识别 subagent 误判（如实际已实现的项标记为缺失）
  - 合并成最终审查报告

报告结构：
  ## 统计表（✅/🔴/🟡/🔵 数量）
  ## 差异明细表（编号/条目/类型/说明/优先级）
  ## 优先级修复清单（P0→P1→P2→P3）
```

### Step 4 — 用户确认

```
呈现 P0/P1/P2/P3 分级清单，请用户逐项确认：
  - ✅ 修复
  - ❌ 跳过
  - ⚠ P3 默认不修
```

### Step 5 — 批量修复

```
按优先级顺序修复：
  P0 阻塞 → 立即单文件改
  P1 重要 → 同一文件多项合并到一个 Edit
  P2 一般 → 批量处理
  P3 可延后 → 默认跳过

原则：
  - 一个文件尽量一次 Edit 覆盖多项
  - 修改完立即 tsc 验证
  - 接口/SDK 用 Toast 占位，注释标注"后续接入"
```

### Step 6 — 验证

```
Bash: npx tsc --noEmit
  exit=0 → 通过
  有报错 → Read 报错文件，修复后重新验证
```

## 四、差异分类标准

| 标记 | 含义 | 处理原则 |
|---|---|---|
| ✅ 一致 | 代码与需求匹配 | 不改 |
| 🔴 缺失 | 需求有但代码无 | 必须补全（P0/P1）或评估后补（P2） |
| 🟡 偏差 | 代码有但实现不同 | 以需求为准修改；若代码实现更优则保留 |
| 🔵 多余 | 代码有但需求无 | 违反需求→删除；合理扩展→保留 |

---

## 五、优先级定义

| 级别 | 标准 | 示例 |
|---|---|---|
| **P0** | 核心流程缺失/运行时错误 | 支付流程缺、硬编码地址、未定义变量 |
| **P1** | 关键字段/校验/流程缺失 | 缺少必填字段、校验正则错误、筛选项缺失 |
| **P2** | 交互细节/视觉/次要功能 | 按钮状态、空状态文案、样式偏差 |
| **P3** | 仅优化建议 | 文案微调、多余 import、dead code |

## 七、已知限制

1. **Subagent API 偶尔 400 错误** —— 此时降级为主对话直接审查（已发生 2 次）
2. **地图/支付 SDK** —— 统一用 Toast/Modal 占位，注释标注"接入 xxx SDK 后替换"
3. **后端接口** —— Mock 数据阶段不校验后端逻辑，仅校验前端交互
4. **uni-app 编译** —— 乘客端无 tsc，仅做代码审查不做编译验证
