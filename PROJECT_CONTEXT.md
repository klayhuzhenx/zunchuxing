## 1. 项目概述

### 1.1 名称与定位

- **项目名**：尊出行
- **业务方**：华为 + 江淮（尊界品牌）+ 和行科技
- **定位**：尊界品牌的高端出行服务平台，覆盖**车主权益用车**与**企业公商务用车**两大场景，做成一套标准化、可交付的服务体系。

### 1.2 用户人群

| 人群 | 来源 | 核心诉求 |
|---|---|---|
| 企业公商务用户 | 企业管理员添加 | 准时、体面，企业结算 |
| 尊界车主（C 端） | 购车赠送年度权益 | 打开就能约，权益自动抵扣 |

### 1.3 系统架构（四端协同）

| 端 | 形态 | 使用者 | 核心职责 |
|---|---|---|---|
| 乘客端 | 微信小程序 | 公商务员工 / 尊界车主 | 下单、查行程、企业管理（管理员）、权益查询 |
| 司机端 | App（原生） | 平台司机 | 接派单、行程执行、收入结算 |
| 企业管理后台 | PC Web | 企业管理员 | 员工 / 用车管理 / 费用结算 |
| 运营管理后台 | PC Web | 尊出行运营 / 财务 / 客服 | 订单调度、车 / 司机管理、企业客户、权益、财务、配置 |

底层能力：调度引擎 ｜ 权益中心 ｜ 支付结算 ｜ 数据看板。

### 1.4 设计规范

- 视觉：尊界品牌色（深色主调 + 金色点缀），HarmonyOS Sans 字体，鸿蒙图标库，8px 栅格，圆角 8-12px。
- 交互：层级 ≤ 3，核心操作 3 步内完成，统一加载/成功/失败反馈，支持大字体 + 屏幕朗读。
- 跨端一致：色彩、字体、图标、组件层面四端统一。

## 2. MVP 范围与业务边界

### 2.1 业务场景（MVP 仅以下两类）

- **包车出行**：长时间 / 长途用车，跨城出差、多站点拜访、全天商务陪同；同司机同车全程。
- **租车出行**：按天计费，自带或附司机租车（详细规则见乘客端文档 §5）。

> **预约出行 / 机场接送**：方案与原型已规划，**MVP 不实现**，后续迭代接入。

### 2.2 用户身份模型

- C 端：尊界车主，享年度权益。
- B 端员工：由企业管理员添加，使用「企业支付」走企业额度。
- 企业管理员：B 端中具备「企业管理」权限的员工，可在乘客端「企业管理」入口或企业 PC 后台维护员工与额度。
- 同一手机号在乘客端用同一账号，同时可绑定 C 端权益和 B 端员工身份。

### 2.3 支付与结算

- 个人支付：微信支付。
- 企业支付：扣减企业总额度，运营后台「调整额度」开通 / 增减；MVP 阶段乘客端 / 企业端**均不开放充值入口**。
- 补款：行程产生超时/超里程，订单流转至「待补款」，乘客通过补款链接支付。
- 退款：取消 / 异常按规则退款，运营触发，财务复核。

### 2.4 权益体系（车主）

- 购车赠送年度权益，详细规则在乘客端「我的 → 权益」中查询。
- 下单时权益自动抵扣，运营后台「运营配置」管理权益模板与发放规则。

## 4. 重要业务约定（决策记录）

> 这些是会话过程中确认的关键决策，新对话务必沿用，避免重复确认。

### 4.1 通用

- MVP **仅包车 + 租车**两种业务，预约 / 机场接送暂不做。
- 乘客端注册登录用**短信验证码 + 极验**，无密码；微信快捷登录需绑定手机号。
- 运营端**不开放注册**，账号由超管创建。

### 4.2 运营端登录

- 接入第三方**极验（GeeTest）**，**每次登录必过**，不再做"5 次失败后启用"。
- 登录错误**统一提示**「账号或密码错误」，避免账号枚举。
- **取消「修改密码」章节**。
- **忘记密码改自助**：账号 + 已绑定手机号 → 极验 + 短信 → 设新密码。

### 4.3 工作台

- **删除「异常订单」核心卡片**（业务上当前没有此逻辑）。
- 待办新增「额度告急企业」入口。

### 4.4 企业客户管理

- 基本信息**仅 4 项**：企业名称 / 统一社会信用代码 / 联系人姓名 / 联系人手机号（营业执照另作图片预览）。
- 额度全部由运营操作；MVP **不开放充值入口**，因此**无「充值」、无「额度预警」**章节。
- 调整额度统一为「调增 / 调减 + 必填原因」，写入额度变动记录。
- 列表筛选**新增**「剩余额度 大于 / 小于 / 等于 + 金额」。
- 「新增企业」弹窗：填四项基本信息 → 状态直接「已通过」→ 以联系人手机号为初始账号自动创建企业管理员 → 短信通知。
- 已禁用：员工乘客端不能用企业支付，**行程中订单照常进行**。

### 4.5 订单管理

- **一级 Tab**：订单 / 司机订单（一笔订单可对应多条司机订单，改派会作废原司机订单）。
- 状态 Tab：全部 / 待支付（含「待补款」子状态）/ 待派车 / 待开始 / 行程中 / 已完成 / 已取消。
- 状态**全部由系统自动流转**，运营**不可手动改状态、不可主动取消订单**。
- 列表 / 详情统一用「派车」按钮，待派车=首派、待开始/行程中=改派。
- 派车弹窗：车辆筛选 + **月历视图**（默认当月、左右切换月份）+ 司机选择。
- 冲突仅红色块**提示**，**允许继续派车**，不强制阻断。
- 车辆**位置信息暂未接入**，列表不显示距离 / 位置。
- 详情新增「下单人信息」「关联司机订单」区块；**乘客备注只读**，运营写「内部备注」。

## 8. 写作约定（保持文档风格一致）

- **语言**：中文（专有名词保留英文）。
- **结构**：每章=业务说明 + 业务流程（Mermaid）+ 列表 / 表单 / 弹窗 / 详情 / 状态规则。
- **流程图**：Mermaid `flowchart TD`，节点 classDef 区分 `step / terminal / warn`。
- **字段表**：列「字段 / 必填 / 校验 / 错误提示」或「列 / 内容」。
- **状态颜色**：待审核=琥珀色 / 已通过=绿色 / 已驳回=灰色 / 已禁用=红色。
- **引号**：业务术语用「」，强调用 **加粗**。
- **避免**：在乘客端文档讨论运营操作、在运营端文档讨论小程序内部逻辑（用跳转引用）。

## 9. 名词表

| 名词 | 含义 |
|---|---|

| B 端 | 企业用户（员工 / 管理员） |
